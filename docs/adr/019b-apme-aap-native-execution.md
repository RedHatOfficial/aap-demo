# ADR-019b: APME AAP-Native Execution

**Status:** Accepted  
**Date:** 2026-07-21  
**Author:** Chad Ferman  
**Supersedes:** ADR-019 Architecture Decision 2 (venv approach)

## Context

ADR-019 established the `apme-eap` addon using official APME playbooks via a bash wrapper with an isolated Python virtual environment. While functional, this approach had drawbacks:

1. **Redundant execution environment**: Created a separate venv when AAP already has execution environments
2. **Missed demo opportunity**: Didn't showcase AAP running automation workloads
3. **Slower first run**: Venv setup takes 1-2 minutes
4. **System dependencies**: Required local Ansible installation
5. **No UI visibility**: Playbook runs were opaque to users

**Key insight:** Since `aap-demo` deploys AAP, we can use AAP's REST API to execute the playbooks *within AAP itself*, eliminating the venv entirely.

## Decision

Replace venv-based `ansible-playbook` execution with **AAP REST API orchestration**. The addon now:

1. Uses AAP's REST API to create/update resources (Project, Inventory, Job Template)
2. Configures Git-based Project pointing to the repository branch
3. Launches playbook execution as an AAP job via REST API
4. Auto-generates service account token and configures job template defaults

### Implementation

### Final Architecture: Git-Based Project with Auto-Configured Defaults

The implementation uses **Git SCM** for project source and **Ansible playbooks** for AAP resource management, avoiding file copy overhead and improving AAP-native integration.

**Ansible Playbooks:**

1. **`playbooks/setup_aap_resources.yml`** - Creates AAP resources via REST API
   - Queries Default organization
   - Creates Git-based Project pointing to current branch
   - Triggers project sync to pull playbook content
   - Creates Inventory (localhost with `ansible_connection: local`)
   - Generates service account token with cluster-admin permissions
   - Creates Job Template with pre-configured defaults:
     - `openshift_api_url: "https://kubernetes.default.svc:443"` (internal cluster address)
     - `openshift_token: <auto-generated>` (service account token)
     - `openshift_project_name: "apme"`
     - All helm chart versions and settings

2. **`playbooks/launch_apme_deployment.yml`** - Executes APME deployment
   - Launches job using pre-configured template defaults
   - Waits for completion with polling
   - Streams job output and error messages
   - Reports success/failure with AAP UI link

3. **`playbooks/deploy_apme_portal.yml`** - Main deployment playbook (runs in AAP)
   - Uses `kubernetes.core` modules with token authentication (no kubeconfig)
   - Deploys portal and APME gateway via Helm charts
   - Direct chart URLs instead of helm CLI repository management
   - Works in AAP execution environments without additional dependencies

**Modified:** `addons/apme-eap/deploy.sh`

- `setup_minimal_venv()` - Creates venv with `ansible-core` (not full Ansible suite)
- `ensure_aap_token()` - Auto-creates token via playbook if not exists
- `deploy()` - Orchestrates playbooks instead of bash functions
- Removed dependency on `lib/aap-api.sh` (bash+curl approach)

**Key Implementation Details:**

```yaml
# Service account token generation (cluster-admin permissions)
- name: Grant cluster-admin to kube-system default service account
  ansible.builtin.command:
    cmd: kubectl create clusterrolebinding apme-deployer-admin --clusterrole=cluster-admin --serviceaccount=kube-system:default

- name: Generate OpenShift service account token
  ansible.builtin.command:
    cmd: kubectl create token -n kube-system default --duration=87600h
```

**Git-based project** (auto-syncs from repository):
```yaml
- name: Create project (Git SCM type)
  ansible.builtin.uri:
    body:
      name: "aap-demo-apme"
      scm_type: "git"
      scm_url: "https://github.com/RedHatOfficial/aap-demo.git"
      scm_branch: "{{ current_branch }}"
      scm_update_on_launch: true
```

**Token authentication without kubeconfig** (critical for AAP execution):
```yaml
# kubernetes.core modules use api_key directly, not kubeconfig
- name: Check if namespace exists
  kubernetes.core.k8s_info:
    kind: Namespace
    name: "{{ openshift_project_name }}"
    api_key: "{{ openshift_token }}"  # No default(omit)!
    host: "{{ openshift_api_url }}"
    validate_certs: false
```

**Helm without helm CLI** (uses repository URLs directly):
```yaml
- name: Install portal Helm release
  kubernetes.core.helm:
    chart_ref: "{{ portal_helm_chart_repo_url }}{{ portal_helm_chart_name }}"
    # NOT: "{{ repo_name }}/{{ chart_name }}"
```

**Job execution flow:**
```
setup_aap_resources.yml (run once locally)
  ↓
1. Create Git-based Project
   - URL: https://github.com/RedHatOfficial/aap-demo.git
   - Branch: current branch (apme-playbook)
   - SCM update on launch: true
  ↓
2. Trigger project sync
   - Wait for successful sync
  ↓
3. Generate service account token
   - Grant cluster-admin to kube-system:default
   - Create long-lived token (10 years)
  ↓
4. Create Job Template with defaults
   - Playbook: addons/apme-eap/playbooks/deploy_apme_portal.yml
   - Pre-configured with token and cluster URL
   - ask_variables_on_launch: true (optional overrides)
  ↓

launch_apme_deployment.yml (from AAP or locally)
  ↓
1. Launch "Deploy APME" job
   - Uses template defaults (no vars needed)
   - POST to /api/controller/v2/job_templates/{id}/launch/
  ↓
2. Poll for completion
   - Check job status every 5 seconds
   - Stream errors on failure
  ↓
3. Display results
   - Success/failure status
   - AAP Web UI link for full job output
```

## Consequences

### Positive

- **Zero manual configuration**: Auto-generates token and configures all job template defaults
- **AAP-native**: Showcases AAP running real automation (great demo story)
- **Git-based**: Uses standard AAP project model (no file copy hacks)
- **Web UI visibility**: Users can watch playbook execution in AAP UI
- **No helm CLI needed**: Uses `kubernetes.core.helm` with direct chart URLs
- **Execution environment ready**: Works in AAP EE without additional dependencies
- **Internal cluster networking**: Uses `kubernetes.default.svc:443` for pod-to-API communication
- **Proper RBAC**: Service account with cluster-admin via ClusterRoleBinding

### Negative

- **Initial complexity**: Token generation and kubernetes.core configuration took iteration
- **RBAC requirements**: Needs cluster-admin permissions (acceptable for demo environment)
- **API surface**: Multiple AAP API endpoints (`/projects/`, `/inventories/`, `/job_templates/`)
- **Debugging**: Must check AAP UI for job logs when troubleshooting

### Neutral

- **Namespace fix**: Also fixed MicroShift compatibility by replacing `Project`/`ProjectRequest` with `Namespace` (separate change in same commit)
- **Same playbooks**: No changes to official APME playbooks themselves

## Alternatives Considered

### 1. Keep venv approach (ADR-019)

**Rejected:** Misses opportunity to showcase AAP's capabilities and creates redundant execution environment.

### 2. Use AWX CLI

**Rejected:** Adds another dependency (`awx` CLI) and doesn't provide more value than direct API calls.

### 3. Git-based Project

Instead of `kubectl cp`, use a git URL pointing to the repo.

**Rejected:**
- Requires git server or public GitHub (aap-demo is often airgapped)
- Adds complexity (SSH keys, credentials)
- `kubectl cp` is simpler for local development

### 4. MCP Server for AAP

Use the (hypothetical) AAP MCP server instead of direct API calls.

**Future consideration:** If AAP MCP server becomes available, this would be cleaner than direct API calls.

## Consequences

### Key Learnings

1. **kubernetes.core auth behavior**: 
   - `api_key: "{{ var | default(omit) }}"` causes module to fall back to kubeconfig when var is undefined
   - Must use `api_key: "{{ var }}"` (no default) to force token-only auth
   - Environment variables (`K8S_AUTH_KUBECONFIG`) must be unset, not set to empty string

2. **Helm without CLI**:
   - `kubernetes.core.helm_repository` requires helm binary
   - Use direct URLs: `chart_ref: "{{ repo_url }}{{ chart_name }}"` instead
   - Works in AAP execution environments without helm CLI

3. **AAP variable passing**:
   - `extra_vars` must be JSON-serialized: `| to_json`
   - Job template defaults eliminate need for launcher to pass variables
   - `ask_variables_on_launch: true` allows optional overrides

4. **Service account tokens**:
   - `kubectl create token` generates time-limited tokens
   - Requires ClusterRoleBinding for cluster-wide permissions
   - Internal cluster address (`kubernetes.default.svc:443`) works from pod-to-pod

## Migration Notes

**From venv-based (ADR-019) to AAP-native (ADR-019b):**

1. No user action required - deployment command is the same
2. Venv now contains `ansible-core` instead of just PyYAML
3. `~/.aap-demo/apme-eap-venv/` is still created but much smaller (~50 MB vs 150+ MB)
4. Token creation is automatic (no manual Web UI steps)
